
# V3 — Idempotent External Execution

## Overview

Version 3 introduces **idempotent execution for external side effects**.

In Version 2, the workflow engine introduced **step leasing** to recover from
worker crashes. While this ensured that steps were eventually executed, it also
introduced the possibility of **duplicate execution**.

If a worker crashes while executing a step, another worker may reclaim the
step and execute it again. This creates an **at-least-once execution model**.

External systems must therefore be able to safely tolerate duplicate requests.

Version 3 solves this problem by introducing **idempotency** in the external
service layer.

---

# Problem in V2

In Version 2 the system guarantees:

steps are eventually executed

However the execution model is:

at-least-once execution

Example failure scenario:

worker A claims step
↓
worker A starts external call
↓
worker A crashes
↓
lease expires
↓
worker B reclaims step
↓
external call executed again

Result:

duplicate external side effects

Examples of problematic side effects:

duplicate email sent
duplicate payment charged
duplicate order created

Therefore the system requires **idempotent external operations**.

---

# Solution

Version 3 introduces an **idempotency layer** in the external service.

Each request contains an **Idempotency-Key** header.

Example request:

POST /execute
Idempotency-Key: step-id

The external service maintains an **idempotency table** that records execution
state and cached responses.

This ensures that duplicate requests return the same result instead of
executing the action again.

---

# Idempotency Table

The external simulator introduces a new table:

idempotency_keys

Example schema:

idempotency_key   TEXT PRIMARY KEY
status            VARCHAR
response_body     JSONB
status_code       INT
created_at        TIMESTAMP

Status values:

IN_PROGRESS
COMPLETED

---

# Execution Flow

## New Request

request arrives
↓
INSERT idempotency_key (status=IN_PROGRESS)
↓
execute external action
↓
store response
↓
update status = COMPLETED

## Duplicate Request

request arrives
↓
idempotency key already exists
↓
check status

If COMPLETED → return stored response

If IN_PROGRESS → return HTTP 409 Conflict

---

# Atomic Insert Pattern

The system relies on the PostgreSQL pattern:

INSERT ... ON CONFLICT DO NOTHING

Example query:

INSERT INTO idempotency_keys
(idempotency_key, status, created_at)
VALUES (:key, 'IN_PROGRESS', now())
ON CONFLICT DO NOTHING

The primary key constraint ensures that only one worker can successfully insert
the idempotency key.

---

# Transaction Model

Database operations are kept short to avoid long-running transactions.

tryInsert() → transaction
execute action → no transaction
complete() → transaction

This ensures external calls do not hold database transactions.

---

# System Guarantees

at-least-once workflow execution
+
idempotent external operations
=
exactly-once side effects

# Key Findings

Duplicate retries do not trigger duplicate execution.

Concurrent requests are safely handled.

Retry storms are mitigated.

PostgreSQL handles contention efficiently.

---

# Limitation Identified

If the external action succeeds but the worker crashes before updating the
idempotency record, the database state becomes:

status = IN_PROGRESS

Future retries see the key as IN_PROGRESS and return HTTP 409 indefinitely.

The system becomes stuck.

---

# Motivation for V4

Version 4 introduces **execution leases** for idempotency records to recover
from abandoned executions.

New fields will include:

lease_expires_at
claimed_by

# Summary

Version 3 introduces idempotent execution for external side effects, ensuring
that duplicate workflow executions do not produce duplicate side effects.

The system now tolerates retries, concurrency, and retry storms safely.

However, a failure scenario exists where idempotency records may become stuck
in IN_PROGRESS after crashes. Version 4 resolves this limitation.
