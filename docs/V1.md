# V1 — Naive Workflow Execution Engine

## Overview

V1 implements a minimal workflow orchestration system capable of:

- Creating workflows
- Persisting ordered steps
- Executing steps sequentially
- Invoking external services
- Tracking workflow completion

The system intentionally starts with a naive design in order to demonstrate reliability issues that appear in distributed systems.

---

## Architecture

Client → Workflow API → PostgreSQL → Worker → External Simulator

---

## Execution Flow

1. Client creates workflow via API
2. Workflow and steps are stored in PostgreSQL
3. Worker polls the database every second
4. First `PENDING` step is selected
5. Step marked `RUNNING`
6. Worker calls external service
7. Step updated to `SUCCESS` or `FAILED`
8. Workflow status updated accordingly

---

## Database Schema

### workflows

| column | description |
|------|-------------|
| id | workflow id |
| status | workflow state |
| created_at | creation timestamp |
| updated_at | last update |
| version | optimistic locking |

### steps

| column | description |
|------|-------------|
| id | step id |
| workflow_id | parent workflow |
| step_order | execution order |
| status | step state |
| external_url | external service endpoint |
| retry_count | retry attempts |
| last_error | error message |
| version | optimistic locking |

---

## Worker Algorithm

poll every second

find first step with status = PENDING

mark step RUNNING

call external service

mark step SUCCESS or FAILED

update workflow state

---

## Known Limitations

V1 intentionally lacks fault tolerance.

Example failure:

Worker crashes while step is RUNNING.

Result:

Step remains RUNNING forever.

Workflow becomes stuck.

This motivates the design of V2, which introduces lease-based execution.
