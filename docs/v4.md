
# V4 — Idempotency Execution Leasing

## Overview
Version 4 introduces execution leasing for idempotent operations.  
While Version 3 ensured idempotent external calls, a failure scenario still existed:

If a worker crashed after executing the external action but before updating the idempotency record, the system could become stuck in `IN_PROGRESS`.

V4 solves this problem by introducing lease-based execution ownership, allowing another worker to reclaim abandoned executions.

---

# Problem in V3

Execution flow in V3:

request arrives  
↓  
idempotency key inserted (`IN_PROGRESS`)  
↓  
external action executed  
↓  
response stored (`COMPLETED`)

Failure scenario:

worker executes external action  
↓  
worker crashes before `COMPLETED` update  
↓  
record remains `IN_PROGRESS` forever  
↓  
future retries return `IN_PROGRESS` indefinitely

System becomes permanently stuck.

---

# Solution

V4 introduces execution leases for idempotency records.

New columns:

- claimed_by
- lease_expires_at

Workers claim execution with a time-limited lease.  
If a worker crashes, the lease eventually expires and another worker can reclaim the execution.

---

# Updated Idempotency Table

Columns:

idempotency_key   TEXT PRIMARY KEY  
status            VARCHAR  
response_body     JSONB  
status_code       INT  
created_at        TIMESTAMP  
claimed_by        TEXT  
lease_expires_at  TIMESTAMP  

---

# Execution Flow

## New Request

request arrives  
↓  
INSERT record (`IN_PROGRESS`)  
↓  
claim execution lease  
↓  
execute external action  
↓  
store response  
↓  
mark `COMPLETED`

---

## Retry Request

request arrives  
↓  
record already exists

Decision:

- COMPLETED → return cached response
- IN_PROGRESS + lease active → return IN_PROGRESS
- IN_PROGRESS + lease expired → reclaim execution

---

# Atomic Lease Reclaim

Lease reclaim uses a single atomic query:

UPDATE idempotency_keys
SET claimed_by = :worker,
    lease_expires_at = :newLease
WHERE idempotency_key = :key
AND lease_expires_at < now()

If the update affects 1 row, reclaim succeeds.  
If 0 rows, another worker already owns the lease.

---

# State Machine

NEW REQUEST  
     │  
     ▼  
INSERT (claim lease)  
     │  
     ▼  
IN_PROGRESS  
     │  
     ├─ success → COMPLETED  
     │  
     └─ crash  
           │  
           ▼  
     lease expires  
           │  
           ▼  
     another worker reclaims

System now provides:

- concurrent reclaim safety
- retry storm resilience

Combined with workflow step leasing, the system ensures eventual exactly-once external side effects.
