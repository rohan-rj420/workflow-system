
# V5 — Retries, Exponential Backoff, and Dead Letter Queue

## Overview
Version 5 introduces retry scheduling, exponential backoff, and a Dead Letter Queue (DLQ) for failed workflow steps.

These improvements allow the workflow engine to tolerate temporary failures in external systems while keeping workers stable.

---

## Problem Before V5

Previously, any external failure resulted in immediate workflow failure.

Worker executes step  
↓  
External API fails  
↓  
Step marked FAILED  
↓  
Workflow marked FAILED

However many external failures are temporary.

Examples:
- Payment provider outage
- Email service downtime
- Network failures
- API rate limits

The system needs retry capability.

---

## Retry Mechanism

When a step fails:

1. Check if `retry_count < max_retries`
2. If true:
   - Increment retry_count
   - Schedule retry with exponential backoff
   - Set `next_retry_at`
3. If false:
   - Move step to Dead Letter Queue

---

## Exponential Backoff

Retry delay grows exponentially.

retry 1 → 2 seconds  
retry 2 → 4 seconds  
retry 3 → 8 seconds  
retry 4 → 16 seconds  

Formula:

delay = 2^retryCount seconds

This prevents retry storms during outages.

---

## Step Lifecycle

PENDING  
↓  
RUNNING  
↓  
FAILED (retry scheduled)  
↓  
FAILED (retry scheduled again)  
↓  
FAILED (retries exhausted)  
↓  
DEAD LETTER QUEUE

---

## Dead Letter Queue

Failed steps exceeding retry limits are stored in:

`dead_letter_steps`

This table contains:
- step id
- workflow id
- retry count
- external URL
- error message
- timestamps

This allows later inspection or replay.

---

## Worker Query Logic

Workers now execute steps that satisfy:

status = 'PENDING'  
OR (status = 'FAILED' AND next_retry_at <= now())  
OR (status = 'RUNNING' AND lease_expires_at < now())

This ensures workers only pick safe, runnable steps.

Version 5 provides:

- stable worker behavior under failures
