# V7 System Design & Architecture (Interview-Grade)

## 🧠 Overview

V7 is a **reliability-first workflow orchestration system** designed to execute multi-step workflows with:
- retries
- failure handling
- concurrency control
- observability

---

## 🧱 Core Architecture

### Components

1. Workflow Engine (Producer)
2. Workflow Worker (Step Claiming)
3. Outbox Dispatcher
4. Result Processor
5. Step Execution Service

---

## 🔁 Execution Flow

1. Workflow created via API
2. Steps stored in DB
3. Worker claims steps (`FOR UPDATE SKIP LOCKED`)
4. Outbox event created
5. Dispatcher executes external call
6. Result stored
7. Result processor updates step
8. Workflow completed when all steps succeed

---

## ⚙️ Concurrency Model

- Worker thread pool (4–8 threads)
- Dispatcher workers with queue
- Backpressure via BlockingQueue

---

## 🗄️ Database Design

### Tables
- workflows
- steps
- outbox_events
- execution_results
- dead_letter_steps

### Patterns Used
- SKIP LOCKED for concurrency
- Idempotency keys
- Retry with exponential backoff
- Lease-based execution

---

## 📊 Observability & Metrics

### Worker Metrics
- Throughput (steps/sec)
- Poll attempts
- Idle polls
- Utilization %
- Queue depth

### Dispatcher Metrics
- Events polled
- Dispatch success/failure
- Dispatch latency

### System Metrics
- DB connection pool (HikariCP)
- JVM (GC, heap)
- HTTP latency
- CPU usage

---

## 🚀 Evolution Summary

### Before Optimization
- Low throughput (~4 steps/sec)
- High latency (up to 150s)
- DB starvation
- System instability

---

### After V7

- Throughput: ~190+ steps/sec
- Latency: ~20ms avg
- Zero failures
- Stable DB utilization
- Predictable performance

---

## 🚨 Limitations

- DB polling overhead
- Tight coupling to DB throughput
- Limited horizontal scalability

---

## 🎯 Why Kafka Next

To solve:
- Polling inefficiency
- DB contention
- Scaling limits

Kafka enables:
- Event-driven execution
- Partitioned scalability
- Near-zero polling cost

---

## 💬 Key Design Principles

- Correctness first
- Idempotency everywhere
- Backpressure over brute force
- Observability-driven optimization
- Incremental evolution

---

## 🧠 Interview Summary

“This system evolved from a failure-prone, DB-bound workflow engine into a stable, high-throughput system by introducing batching, backpressure, and query optimization. Once the system became DB-efficient, further scaling required moving to an event-driven architecture.”
