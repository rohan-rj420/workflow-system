
# V2 — Lease-Based Fault-Tolerant Workflow Execution

## Overview

Version 2 introduces **lease-based step execution** to make the workflow engine fault tolerant.

In V1, workers executed steps directly after polling the database. If a worker crashed during execution, the step would remain stuck in the `RUNNING` state indefinitely.

V2 solves this by introducing **step leasing**, allowing other workers to reclaim steps when the original worker fails.

The system now guarantees **at-least-once execution** of workflow steps.

---

# Problem in V1

The V1 architecture used a simple polling worker model:

poll database  
↓  
select PENDING step  
↓  
mark step RUNNING  
↓  
execute external call  
↓  
mark SUCCESS / FAILED  

This design had a major reliability issue.

If a worker crashed after marking a step `RUNNING`, the step would remain permanently stuck because no other worker would attempt to execute it again.

Example failure scenario:

worker claims step  
↓  
worker crashes  
↓  
step remains RUNNING forever  
↓  
workflow never completes  

This violates a key requirement of reliable workflow systems: **tasks must not be permanently lost due to worker failures**.

---

# V2 Design — Step Leasing

V2 introduces **lease-based task ownership**.

Each step now contains two new fields:

- claimed_by  
- lease_expires_at  

These fields allow workers to temporarily claim steps.

### Step Lifecycle

PENDING  
↓ claim  
RUNNING (claimed_by = worker)  
↓ success  
SUCCESS  

If a worker crashes before completing the step:

RUNNING (lease expires)  
↓  
another worker reclaims step  
↓  
execution continues  

---

# Database Changes

The following columns were added to the `steps` table:

- claimed_by TEXT  
- lease_expires_at TIMESTAMP  

These fields track the current worker that owns the step and when the claim expires.

An index was added to improve worker polling performance:

CREATE INDEX idx_steps_claimable  
ON steps (status, lease_expires_at, created_at);

This allows workers to efficiently find reclaimable tasks.

---

# Atomic Task Claiming

Workers now claim steps using an atomic query:

SELECT *  
FROM steps  
WHERE  
    status = 'PENDING'  
    OR (status = 'RUNNING' AND lease_expires_at < now())  
ORDER BY created_at  
FOR UPDATE SKIP LOCKED  
LIMIT 1  

This query ensures multiple workers can safely compete for tasks.

The `SKIP LOCKED` clause prevents workers from selecting rows that are already locked by another transaction.

This allows safe concurrent task processing across multiple worker instances.

---

# Adaptive Polling

To reduce unnecessary database load when the queue is empty, V2 introduces **adaptive polling**.

Worker polling delay increases gradually when no work is available.

Polling behavior:

initial delay = 100 ms  
no step found → delay += 100 ms  
step found → reset delay to 100 ms  

This ensures:

- high responsiveness when queue is busy  
- reduced database load when queue is empty

# Experimental Validation

Several experiments were conducted to validate the V2 design.

---

# Experiment 1 — Multi-Worker Concurrency

## Objective

Verify that multiple workers can execute steps concurrently.

## Setup

Two workflow-engine instances were started locally.

Workers compete for steps using the atomic claim query.

## Observed Metrics

Worker metrics from both instances:

Throughput (steps/sec): ~4.4  
Poll attempts: ~22  
Idle polls: 0  
Worker utilization: 100%  

Database state during execution:

status | count  
------ | -----  
RUNNING | 2  
SUCCESS | 2194  
PENDING | 7672  

## Result

RUNNING steps ≈ number of workers.

This confirms that workers process steps concurrently.

---

# Experiment 2 — Worker Crash Recovery

## Objective

Verify that steps are reclaimed when a worker crashes.

## Procedure

1. Two workers were started.
2. A workflow step was claimed by worker A.
3. Worker A process was terminated.
4. After lease expiration, worker B reclaimed the step.

## Observed Database State

Initial claim:

status: RUNNING  
claimed_by: worker-e6744ce6  
lease_expires_at: 15:24:48  

After worker crash and lease expiry:

status: RUNNING  
claimed_by: worker-8bd30757  
lease_expires_at: 15:24:59  

Final state:

no RUNNING steps  
step transitioned to SUCCESS  

## Result

The step was successfully reclaimed and executed by another worker.

This confirms that V2 prevents tasks from being permanently lost due to worker crashes.

---

# Experiment 3 — Duplicate Execution

## Objective

Investigate behavior when a worker crashes during step execution.

## Observed Logs

Worker A:

worker-e6744ce6 claimed step 14ce6cc9...  
worker-e6744ce6 executing step 14ce6cc9...  

Worker A was terminated before completing the step.

Worker B:

worker-8bd30757 claimed step 14ce6cc9...  
worker-8bd30757 executing step 14ce6cc9...  

Both workers executed the same step.

## Result

The external service received **two identical HTTP requests**.

This confirms that the system provides **at-least-once execution semantics**.

---

# Experiment 4 — Worker Throughput

Worker throughput stabilized around:

~4 – 4.4 steps/sec per worker

External step execution time:

~200 ms

Theoretical maximum:

1 / 0.2s ≈ 5 steps/sec

Observed throughput is close to the theoretical limit, indicating minimal system overhead.

---

# Key Findings

The V2 architecture successfully introduces fault tolerance and safe concurrent execution.

Key improvements over V1:

- atomic task claiming  
- safe multi-worker concurrency  
- lease-based crash recovery  
- adaptive polling  

However, experiments revealed an important limitation.

The system guarantees **at-least-once execution**, but does not prevent duplicate execution when workers fail during step processing.

---

# Limitations of V2

Failure scenario:

worker executes external API  
external API succeeds  
worker crashes before marking SUCCESS  
lease expires  
another worker reclaims step  
external API executed again  

This results in duplicate external requests.

---

# Next Evolution — V3

To safely handle duplicate execution, the system must introduce **idempotency**.

V3 will implement **idempotency keys**, allowing external systems to safely ignore duplicate requests.
