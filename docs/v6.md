# V6 -- Event Driven Execution with Outbox Dispatcher

## Overview

V6 introduces a major architectural evolution of the workflow system.\
Instead of executing external calls directly inside the workflow engine,
execution is now decoupled through the Outbox Pattern.

The system now consists of three cooperating components:

1.  Workflow Engine
2.  Outbox Dispatcher
3.  External Simulator

Execution becomes asynchronous, reliable, and horizontally scalable.

------------------------------------------------------------------------

# Architecture

Workflow Engine responsibilities:

-   Claim runnable steps
-   Publish execution requests to the outbox table
-   Process execution results
-   Apply retry policies and DLQ rules

Outbox Dispatcher responsibilities:

-   Poll pending outbox events
-   Dispatch external calls
-   Write execution results

External Simulator responsibilities:

-   Execute business logic
-   Enforce idempotency
-   Return deterministic results

------------------------------------------------------------------------

# Core Tables

## Outbox Events

Stores events generated by the workflow engine.

Statuses:

PENDING → PROCESSING → PROCESSED\
PENDING → PROCESSING → FAILED

Used by dispatcher workers.

------------------------------------------------------------------------

## Execution Results

Stores results of external calls.

Fields include:

-   step_id
-   workflow_id
-   success
-   error
-   processed flag

These results are later consumed by the workflow engine.

------------------------------------------------------------------------

# Event Driven Flow

Step execution now follows this flow:

Worker claims step\
↓\
Workflow Engine publishes STEP_EXECUTION_REQUESTED event\
↓\
Outbox Dispatcher polls event\
↓\
External service invoked\
↓\
Execution result written\
↓\
Workflow Engine processes result\
↓\
Step marked SUCCESS / FAILED

------------------------------------------------------------------------

# Reliability Guarantees

## Outbox Pattern

Ensures no execution request is lost even if the workflow engine
crashes.

Events are stored in the database before dispatch.

## Idempotent Execution

External calls include an idempotency key equal to step_id.

Duplicate dispatches are safely ignored.

## Retry & DLQ Interaction

If result processing indicates repeated failure:

retry_count increases\
backoff applied\
eventually step moves to DLQ.

------------------------------------------------------------------------

# Concurrency Control

Dispatchers use:

SELECT ... FOR UPDATE SKIP LOCKED

This allows multiple dispatchers to safely process the same outbox
table.

------------------------------------------------------------------------

# Backpressure

Backpressure is achieved using:

ThreadPoolExecutor in dispatcher

This limits the number of concurrent external calls.

Even under burst load the dispatcher cannot exceed configured
concurrency.

------------------------------------------------------------------------

# Handling Late Results

During experiments a problem appeared:

Late arriving execution results after a step had already been moved to
DLQ.

This produced exceptions:

NoSuchElementException in StepExecutionService.

Solution:

ResultProcessingService now safely ignores results for missing steps.

------------------------------------------------------------------------

# Handling Out-of-Order Results

Because dispatchers operate concurrently, results can arrive out of
order.

The system remains correct because:

-   Step state transitions are idempotent
-   Execution results are processed transactionally
-   DLQ removal deletes steps safely
